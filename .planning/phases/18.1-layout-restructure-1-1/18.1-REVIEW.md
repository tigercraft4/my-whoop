## Findings

### WR-001 — `DateFormatter` criado dentro de propriedades computed (HomeView, HealthView)
**Severity:** Warning
**File:** `ios/OpenWhoop/Tabs/HomeView.swift:43–46` e `HealthView.swift:47–51`
**Issue:** `DateFormatter` é instanciado dentro de `dateLabelText` (computed var) e `isoFormatter()` (função chamada 3x em series helpers). `DateFormatter` é uma classe pesada — criar instâncias por render cycle causa overhead de performance. Em HealthView, `isoFormatter()` é chamado 3 vezes por acesso às series computed vars.
**Recommendation:** Extrair como `private static let` ou `private let` na struct (Swift struct lazy init via `static let` é thread-safe):
```swift
// HomeView
private static let dateFmt: DateFormatter = {
    let f = DateFormatter()
    f.dateFormat = "EEE d MMM"
    f.locale = Locale(identifier: "pt_PT")
    return f
}()

// HealthView
private static let isoFmt: DateFormatter = {
    let f = DateFormatter()
    f.dateFormat = "yyyy-MM-dd"
    return f
}()
```

### WR-002 — `@AppStorage("hk.authDeniedShown")` em HomeView — variável não utilizada
**Severity:** Warning
**File:** `ios/OpenWhoop/Tabs/HomeView.swift:18`
**Issue:** `@AppStorage("hk.authDeniedShown") private var authDeniedShown = false` é declarado mas nunca lido nem escrito em HomeView. A lógica de HealthKit auth permanece em TodayView (correcto — HomeView não faz auth). Esta declaração desnecessária cria confusão sobre onde a lógica de auth existe.
**Recommendation:** Remover a declaração de HomeView — não é utilizada e é enganosa.

### WR-003 — EnvironmentObject `hkExporter` injectado em HomeView mas não utilizado
**Severity:** Warning
**File:** `ios/OpenWhoop/Tabs/HomeView.swift:14`
**Issue:** `@EnvironmentObject private var hkExporter: HealthKitExporterViewModel` é declarado mas nenhuma propriedade ou método de `hkExporter` é referenciado no corpo de HomeView. O object é injectado pela cadeia de NavigationStack mas nunca consultado.
**Recommendation:** Remover a declaração `@EnvironmentObject private var hkExporter`. Se HomeView precisar de HealthKit no futuro, readicionar então. Actualmente é dead code que força o caller a injectar um EnvironmentObject desnecessário.

### IN-001 — `isLoading` em HealthView não reflectido na UI
**Severity:** Info
**File:** `ios/OpenWhoop/Tabs/HealthView.swift:16`
**Issue:** `@State private var isLoading = false` é definido e actualizado em `loadWeeklyMetrics()`, mas nunca lido no `body`. O utilizador não vê indicação visual de carregamento enquanto os dados de 7 dias são buscados.
**Recommendation:** Adicionar indicador de carregamento condicional no body:
```swift
if isLoading && weeklyMetrics.isEmpty {
    ProgressView()
        .frame(maxWidth: .infinity, minHeight: 200)
        .tint(WH.Color.textSecondary)
}
```
Ou simplificar removendo `isLoading` completamente e usando o estado vazio das series (MetricChart já mostra "Not enough data" quando `series.count < 2`).

### IN-002 — Botão W overlay pode sobrepor-se à tab bar em alguns dispositivos
**Severity:** Info
**File:** `ios/OpenWhoop/App/RootTabView.swift:50–52`
**Issue:** O botão W usa `.padding(.bottom, 8)` fixo. Em iPhones com Home Indicator (iPhone X+), a tab bar tem altura variável (~83pt). O padding de 8pt pode colocar o botão W parcialmente sobreposto ao safe area inferior em alguns modelos ou orientações.
**Recommendation:** Considerar usar `GeometryReader` ou `.safeAreaInset(edge: .bottom)` para posicionar o botão acima da tab bar de forma adaptativa. Alternativa simples: aumentar para `.padding(.bottom, 16)` que funciona na maioria dos casos.
