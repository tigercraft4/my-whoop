<objective>
Wire the new `/v1/today` endpoint into the iOS sync stack. Add `ServerSync.getTodayMetric()` that fetches and parses the single most-recent `DailyMetric` from `/v1/today`. Update `pullDerived()` to also call this endpoint so the most-recent daily row is always populated after a sync — even when it falls outside the `derivedWindowDays=60` window. Confirm server-values have precedence over `LocalMetricsComputer` (already the case via the existing `pullDerived` → `upsertDailyMetrics` ON CONFLICT DO UPDATE path; verify and document).
</objective>

<tasks>

<title>Add getTodayMetric() to ServerSync.swift</title>
<read_first>
- ios/OpenWhoop/Upload/ServerSync.swift — `getDaily(from:to:)` (line ~343) as the reference for the HTTP GET + JSON parsing pattern; `DailyMetric` construction with field mapping (recovery normalization `/ 100.0`); `get(path:)` private helper; `ServerSync.int` and `ServerSync.dbl` coercion helpers
</read_first>
<action>
Add a new private method `getTodayMetric() async -> DailyMetric?` to `ServerSync` immediately after the `getDaily(from:to:)` method.

Implementation:
- Path: `/v1/today?device=\(deviceId)` — single query parameter, no date range
- HTTP GET via `await get(path: path)` — same private helper as `getDaily`
- JSON parse: `try? JSONSerialization.jsonObject(with: body)` → cast to `[String: Any]`
- If result is `NSNull` or body is `"null"`, return `nil` (server returns JSON null when no rows)
- Build `DailyMetric` using the same field mapping as in `getDaily`: snake_case keys, recovery `/ 100.0` normalization, `ServerSync.int` / `ServerSync.dbl` coercion
- Return `nil` on any network error, parse failure, or missing `day` field
</action>

- Swift file compiles without errors (verified via xcodebuild or swift build)
</acceptance_criteria>
</task>

<task id="T2" type="execute">
<title>Call getTodayMetric() inside pullDerivedWindow()</title>
<read_first>
- ios/OpenWhoop/Upload/ServerSync.swift — `pullDerivedWindow(days:)` implementation; `store.upsertDailyMetrics(_:deviceId:)` call pattern (the existing ON CONFLICT DO UPDATE upsert that makes server win over local)
</read_first>
<action>
In `pullDerivedWindow(days:)`, after the existing `store.upsertDailyMetrics(days, deviceId:)` call, add:

```swift
// Ensure the most-recent daily row is always present — /v1/today uses ORDER BY day DESC LIMIT 1
// so it returns the latest row even if it falls outside the derivedWindowDays window.
if let todayMetric = await getTodayMetric() {
    try? await store.upsertDailyMetrics([todayMetric], deviceId: deviceId)
}
```

This is additive — it does not change the existing window pull. It guarantees that even if the user's device clock or the server's most-recent day falls outside the 60-day window, the single most-recent row is always synced.
</action>
<acceptance_criteria>
- `pullDerivedWindow` contains a call to `await getTodayMetric()`
- The result is upserted via `store.upsertDailyMetrics([todayMetric], deviceId:)` — same upsert method as the window pull
- The call is guarded with `if let todayMetric` — nil is handled gracefully (no crash when server has no data)
- The existing window pull (`guard let days = await getDaily(...)`) is unchanged
</acceptance_criteria>
</task>
