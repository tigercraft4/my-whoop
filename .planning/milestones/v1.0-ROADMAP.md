# Milestone v1.0: WHOOP 5.0 Protocol + iOS App

**Status:** ✅ SHIPPED 2026-05-31
**Phases:** 1–5
**Total Plans:** 21

## Overview

A complete port of the WHOOP 4.0 reverse-engineering project to WHOOP 5.0 hardware. The milestone delivered three things: (1) a fully characterised 5.0 BLE protocol, (2) a canonical decode schema and Python decoder, and (3) a functional iOS app connecting to the WHOOP 5.0 end-to-end, backed by an optional self-hosted server.

The defining discovery was the Maverick outer wrapper: WHOOP 5.0 uses the legacy 4.0 inner frame format (0xAA SOF, command/event enums) for writes (phone→WHOOP) but wraps all responses (WHOOP→phone) in a Maverick envelope (0xAA 0x01, len u16 at buf[2..3], role byte, 3-byte token, type at body[4]). This asymmetric framing was not documented anywhere and required systematic reverse-engineering.

## Milestone Summary

### Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| Clean fork (not dual-support) | Protocol may differ enough to pollute 4.0 codebase | ✓ Correct — Maverick wrapper required new strip_maverick path |
| Mac PacketLogger as primary capture | No jailbreak, captures full HCI post-decryption | ✓ Correct — produced all corpus needed for framing RE |
| Skip RF sniffer (nRF52840) | HCI logs give decrypted GATT | ✓ Correct — unnecessary |
| Phase 3 as explicit CRC gate | All decode work wasted if framing wrong | ✓ Critical — 0% pass rate triggered Maverick RE |
| Python discovery before Swift | Swift iteration 10–100× slower for byte-level RE | ✓ Correct |
| Maverick write/read asymmetry | WHOOP 5.0 accepts 4.0 writes, sends Maverick reads | ✓ Resolved in Phase 5 (D-11) |

### Issues Resolved

- Maverick outer wrapper fully characterised (Phase 3) — unknown pre-project
- 4.0 CRC gate 0% pass rate explained by Maverick wrapper (Phase 3)
- D-11: Asymmetric write/read framing discovered and resolved (Phase 5)
- CBATTError.insufficientEncryption → iOS SMP pairing → bond retry sequence (Phase 5)
- willRestoreState bug: do NOT call discoverServices() before centralManagerDidUpdateState(.poweredOn) (Phase 5, commit dc3e5cf)
