## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] venv must use python3.11, not the plan's `python3`**
- **Found during:** Task 1 verification setup
- **Issue:** System `python3` is 3.9.6, which cannot import the package (`int | None` PEP-604 requires 3.10+) — the plan's `python3 -m venv` command fails to run the decoder.
- **Fix:** Used `python3.11 -m venv /tmp/whoop_venv5` (a pre-existing, documented 05-01 environment constraint). No code change.
- **Files modified:** none.

**2. [Rule 1 - Bug] `_read()` raised `KeyError` on `dtype: "bytes"` (COMMAND_RESPONSE payload)**
- **Found during:** Task 1 (first generator run)
- **Issue:** The 5.0 schema's COMMAND_RESPONSE `payload` field has `dtype: "bytes"`. The shared Python `_read()` looked it up in `_DTYPE` and crashed; the Swift `readDType` already returns nil for unknown dtypes, so the decoders diverged (and Python couldn't decode any COMMAND_RESPONSE).
- **Fix:** `_read()` now returns `None` for unknown scalar dtypes (the static-field walk skips them; bytes/ascii regions are surfaced by post-hooks), mirroring Swift.
- **Files modified:** server/packages/whoop-protocol/whoop_protocol/interpreter.py
- **Commit:** 576fb44

**3. [Rule 1 - Bug] Non-deterministic type-48 resolution (EVENT vs EVENT_BATTERY_LEVEL)**
- **Found during:** Task 1 (schema inspection)
- **Issue:** The 5.0 schema defines two packets at type 48; dict-insertion order decided the winner. Python and Swift `[String:PacketSpec]` iteration orders differ (and Swift's varies per process), so type-48 frames would decode differently across the two languages / across runs — the parity gate would be flaky. `EVENT_BATTERY_LEVEL` also references a non-existent `event_battery` post-hook.
- **Fix:** Both decoders now index `byType` in sorted-name order with first-claim-wins, so `EVENT` (whose `event` post-hook correctly branches on the event number, including BATTERY_LEVEL — the 4.0 pattern) deterministically wins type 48.
- **Files modified:** schema.py, Schema.swift
- **Commit:** 576fb44

**4. [Rule 3 - Blocking] Deferred 4.0 parity/schema test breakage (8 suites)**
- **Found during:** Task 2 (`swift test`)
- **Issue:** Switching the runtime schema to 5.0 in 05-01 invalidated every 4.0-fixture/4.0-schema test (`ParityTests`, `StreamsParityTests`, `StreamsTests`, `SchemaTests`, `InterpreterEnvelopeTests`, `HistoricalV24Tests`, `HistoricalStreamsParityTests`, `BiometricStreamsParityTests`). The plan's acceptance criterion requires `swift test` to exit 0. The 5.0 schema has no type-47 V24 / type-43 optical layouts, and fabricating them is forbidden (Pitfall 5; 4.0 is out of scope for this fork per PROJECT.md).
- **Fix:** Added a test-only `overrideSchemaResource("whoop_protocol")` hook to `loadSchema()` (production default unchanged at `whoop_protocol_5`) and pinned each 4.0-protocol test to the 4.0 schema in `setUp`/`tearDown`. The 4.0 decode path stays validated; no 5.0 offsets fabricated. Logged the resolution in `deferred-items.md`.
- **Files modified:** Schema.swift + the 8 test files + deferred-items.md
- **Commit:** 0ce7c4d
